## Context

全新项目 Carbon Point——多租户 SaaS 碳积分打卡平台。目标用户为有健康管理需求的企业，通过爬楼梯打卡送积分的机制激励员工运动，积分可兑换虚拟商品。平台以多租户形式交付，每个注册企业拥有独立的规则配置、商品管理、积分运营体系。

前端需嵌入微信小程序 WebView 和已有 APP WebView，采用 H5 方案实现跨端适配。后端为 Spring Boot 3.x 单体应用，未来可拆分微服务。

## Goals / Non-Goals

**Goals:**

- 多租户 SaaS 架构，企业数据逻辑隔离（tenant_id 字段级）
- 灵活的积分规则引擎，支持时段规则、连续打卡奖励、特殊日期翻倍、等级系数、每日上限
- RBAC 权限体系，每企业独立角色配置，菜单+按钮+API 三级控制
- 虚拟商品商城，支持券码、直充、权益三种商品类型
- 三端前端应用：用户端 H5、企业管理后台、平台运营后台
- 平台级企业管理与数据看板

**Non-Goals:**

- 真实步数/传感器数据采集（当前为随机积分打卡，非真实运动数据）
- 实物商品与物流体系（当前仅虚拟商品）
- 微信小程序原生开发（使用 H5 WebView 嵌套）
- 支付系统（纯积分兑换，不涉及金额交易）
- 企业间数据隔离升级为物理隔离（当前逻辑隔离足够）
- 国际化与多语言（中文为主）
- 移动端原生 APP 开发

## Decisions

### 1. 多租户隔离方案：共享数据库 + tenant_id

**选择**: 所有租户共享一个 MySQL 数据库，通过每张业务表的 tenant_id 字段实现逻辑隔离。

**理由**:
- 初期企业数量预计在几百以内，逻辑隔离足够
- 运维成本最低，单库备份、监控、升级
- MyBatis-Plus 的 TenantLineInnerInterceptor 成熟可用，自动拼装 WHERE tenant_id = ?
- 后续如有需要可迁移到 Schema 隔离或独立库

**备选方案**:
- 每企业独立数据库：数据物理隔离更安全，但运维复杂度高，当前规模不需要
- Schema 隔离（同一 MySQL 实例多 Schema）：折中方案，但 MyBatis-Plus 支持不如字段隔离成熟

### 2. 前端架构：pnpm Monorepo

**选择**: pnpm workspace，apps/h5（用户端）+ apps/dashboard（企业后台+平台后台合并）+ packages/（共享）。

**理由**:
- 企业后台和平台后台 UI 结构类似（都是后台管理面板），合并一套代码通过登录身份区分菜单
- H5 交互模式完全不同（移动端、打卡动画），独立应用更灵活
- 共享 packages 避免重复（API 层、组件库、hooks、工具函数）

### 3. 积分规则引擎：JSON 配置 + 规则链执行

**选择**: point_rules 表存储规则配置，type 字段区分规则类型，config JSON 字段存储每种类型的具体参数。打卡时按规则链顺序执行计算。

**理由**:
- JSON config 灵活，不同规则类型可以存储不同结构的参数，无需频繁改表
- 规则链模式可扩展，新增规则类型只需新增 type 枚举和对应的计算逻辑
- 时段规则做防重叠校验，保存时检查时间区间是否与已有规则冲突

**规则执行顺序（固定不变，不得调整）**:
1. 时段匹配 (time_slot match) → 确定基础随机积分
2. 特殊日期倍率 (special_date multiplier) → 乘以倍率（如有）
3. 等级系数 (level coefficient) → 乘以等级系数（等级放大）
4. 四舍五入取整 (rounding)
5. 每日上限检查 (daily cap) → 在等级放大之后截断（**每日上限仅针对步骤 1-4 的单次打卡积分**）
6. 连续打卡奖励 (consecutive reward) → 最后单独发放（**不计入每日上限，独立额外计算**）

> **详细定义见 point-engine/spec.md 中的 "积分计算链执行顺序（固定不变）" 章节。**

### 4. RBAC 权限：用户→角色→权限，每租户独立

**选择**: 标准 RBAC 模型，roles / role_permissions / user_roles 三张表，每租户独立角色定义。

**理由**:
- 权限树按模块划分（dashboard/member/rule/product/order/point/report），每个模块下细分操作权限
- 预设角色模板（超管/运营/客服/商品管理/只读），企业开通时自动创建
- 支持企业自定义角色，超级管理员可创建/编辑/删除自定义角色
- 安全约束：每企业至少保留一个超级管理员，最后一个超管不可被降级或删除
- 前端根据权限列表动态渲染菜单和按钮，后端 API 层做权限校验拦截

### 5. 虚拟商品类型与核销

**选择**: products 表的 type 字段区分商品类型，fulfillment_config JSON 存储发券/充值/权益的具体配置。

**理由**:
- **券码型**（coupon）：兑换后生成唯一券码，用户出示或复制使用，管理员后台或扫码核销
- **直充型**（recharge）：兑换时用户填写手机号，系统对接充值接口（可后期扩展）
- **权益型**（privilege）：兑换后直接激活某种权益（如 VIP 会员 N 天）
- 统一由 exchange_orders 表管理订单状态流转：pending → fulfilled → used / expired / cancelled

### 6. 用户注册与绑定

**选择**: 两种路径——企业邀请链接 + 管理员批量导入，一人仅属一企业。

**理由**:
- 邀请链接：tenant_invitations 表管理邀请码、有效期、使用次数，用户打开 H5 注册时自动绑定企业
- 批量导入：管理员上传 Excel（手机号+姓名），系统批量创建账号并绑定企业，batch_imports 记录导入结果
- 用户模型 1:1 绑定企业，简化数据隔离和权限判断

### 7. 认证方案：JWT

**选择**: Spring Security + JWT Token，登录后签发 access_token（短效）+ refresh_token（长效）。

**理由**:
- H5/小程序/APP 均适合 Token 认证
- JWT payload 携带 user_id + tenant_id + roles，减少数据库查询
- Redis 存储 refresh_token 做续期和主动失效
- 平台管理员与企业用户分两套认证逻辑（不同登录入口、不同权限体系）

### 8. 登录安全增强：多层防护体系

**选择**: 图形验证码 + 密码策略 + 登录限流 + 安全日志 + 异常检测的多层安全防护。

**理由**:
- **验证码**: 图形验证码防自动化暴力破解，滑动验证码提升平台管理员登录安全等级
- **密码策略**: Argon2 加密替代 BCrypt，密码强度实时校验，历史密码记录防止复用
- **登录限流**: IP/账号双维度限流，自动锁定机制，有效阻止暴力破解
- **安全日志**: 完整记录所有登录尝试，支持安全审计和事后追溯
- **异常检测**: 新设备/异地/异常时间登录检测，及时发现账户盗用风险
- **配置化**: 所有安全策略支持配置化调整，适应不同安全等级需求

**备选方案**:
- 仅依赖验证码：防护不足，仍可能被绕过
- 仅依赖 MFA：用户体验较差，普通用户接受度低
- 第三方安全服务：增加外部依赖和成本，当前阶段自研足够

**实现优先级**:
- P0: 图形验证码、登录限流、密码强度校验
- P1: 安全日志、Argon2 加密、忘记密码优化
- P2: 滑动验证码、密码历史、异常登录检测

### 9. 验证码方案：图片验证码为主，滑动验证码可选

**选择**: 普通用户登录使用图片验证码，平台管理员可选滑动验证码。

**理由**:
- **图片验证码**: 实现简单，用户体验好，能防御大部分自动化攻击
- **滑动验证码**: 安全性更高，但实现复杂度大，仅用于高权限账号
- **Redis 存储**: 验证码 5 分钟过期，验证后立即删除，防止重用
- **排除易混淆字符**: 0/O/1/I/l 等字符不使用，降低用户输入错误率

### 10. 密码加密：Argon2id 算法

**选择**: 使用 Argon2id 替代 BCrypt 作为密码哈希算法。

**理由**:
- Argon2 是 Password Hashing Competition 获胜算法，安全性经业界验证
- Argon2id 兼顾抵抗 GPU 攻击和侧信道攻击
- 可配置内存占用、并行度、迭代次数，灵活平衡安全与性能
- Spring Security 6+ 原生支持 Argon2，集成成本低

**参数配置**:
- salt-length: 16 bytes
- hash-length: 32 bytes
- parallelism: 4
- memory: 65536 KB (64MB)
- iterations: 3

### 11. 登录限流：双维度计数 + 自动锁定

**选择**: IP + 账号双维度登录失败计数，达到阈值自动锁定。

**理由**:
- **IP 维度**: 防止来自同一 IP 的批量暴力破解
- **账号维度**: 防止针对特定账号的定向攻击
- **Redis 存储**: 原子计数器 + 过期时间，实现简单高效
- **分级策略**: 失败 3 次要求验证码，失败 5 次锁定 30 分钟，平衡安全与体验

**限流规则**:
- 时间窗口: 5 分钟
- IP 阈值: 5 次 → 锁定 30 分钟
- 账号阈值: 5 次 → 锁定 30 分钟
- 验证码触发: 任意维度失败 3 次

### 12. 忘记密码：多因素验证 + 安全重置

**选择**: 短信/邮箱验证码验证身份，重置后强制清除所有会话。

**理由**:
- **多种验证方式**: 支持短信/邮箱/安全问题，适应不同用户场景
- **Token 机制**: 验证成功后签发一次性 resetToken，15 分钟有效
- **会话清除**: 密码重置后立即失效所有 refresh_token，防止已泄露 Token 被滥用
- **密码策略**: 重置时需满足最新密码强度要求

## Risks / Trade-offs

- **[租户数据泄露]** → MyBatis-Plus 拦截器自动拼 tenant_id，但需注意：原生 SQL、跨租户查询（平台管理员看全企业数据）需绕过拦截器。方案：平台管理员的查询使用 @InterceptorIgnore 注解跳过租户拦截，手动在 Service 层做权限校验。
- **[积分通胀]** → 每日积分上限可控制，但企业管理员可能配置过于宽松的规则。方案：平台可设置默认推荐规则模板，并在平台看板中提供积分发放量预警。
- **[规则引擎复杂度]** → JSON config 灵活但查询不便。方案：规则数据量小（每企业通常不超过几十条），加载到内存执行，不影响性能。
- **[打卡并发]** → 同一时段可能存在用户短时间内重复打卡请求。方案：数据库唯一索引（user_id + 打卡日期 + 时段规则 ID）+ 分布式锁（Redis）双重保障。
- **[H5 WebView 兼容性]** → 不同小程序/APP 的 WebView 内核版本不同。方案：使用 Babel 转译确保兼容性，避免使用过新的 JS API，做好降级处理。
- **[验证码被识别]** → OCR 技术可能识别简单图形验证码。方案：增加干扰线和干扰点，使用混合字体，重要场景使用滑动验证码；定期评估验证码安全性，必要时升级验证码方案。
- **[密码加密性能]** → Argon2 内存占用较高，大量用户同时登录可能影响性能。方案：配置合理的 Argon2 参数（memory=64MB, parallelism=4），登录接口独立线程池隔离，监控登录响应时间。
- **[登录限流误伤]** → 同一企业用户共享出口 IP，可能导致正常用户被误伤锁定。方案：IP 锁定阈值适当放宽（5 次/5分钟），提供自助解锁功能（通过手机/邮箱验证），管理员后台可手动解锁。
- **[会话安全]** → refresh_token 被盗用可能导致账户被持久控制。方案：refresh_token 每次使用后轮换，绑定设备指纹，异常设备登录需二次验证，密码修改后立即失效所有会话。
- **[验证码短信成本]** → 大量用户使用短信验证码重置密码可能产生较高成本。方案：优先推荐邮箱验证，设置短信验证码频率限制（同一手机号 1 分钟 1 次，1 小时 5 次），提供图形验证码防止刷短信。

## 🚨 风险缓解计划

### 风险 1：数据不一致导致业务问题
**概率**：中  
**影响**：高  
**缓解措施**：
- 新系统仅用于读取，不处理写入
- 保持旧系统作为主要数据源
- 实时监控数据一致性，发现问题立即告警
- 有快速回滚机制（1分钟内切换回旧系统）
- 先在非生产环境充分验证

### 风险 2：新 API 性能不达标
**概率**：中  
**影响**：中  
**缓解措施**：
- 提前进行性能基准测试
- 保留旧 API 作为 fallback
- 优化查询（索引、缓存、N+1 问题）
- 考虑使用 Redis 缓存热点数据
- 性能不达标时暂停灰度发布

### 风险 3：团队进度延期
**概率**：中  
**影响**：中  
**缓解措施**：
- 每日站会跟踪进度
- 关键任务设置缓冲区
- 优先保证核心功能，次要功能可延后
- 必要时增加资源（加班或调人）
- 及时调整计划，与利益相关者沟通

### 风险 4：功能开关设计不完善
**概率**：低  
**影响**：高  
**缓解措施**：
- 开关设计经过架构评审
- 支持多维度开关（全局、租户、用户）
- 开关变更有审计日志
- 定期清理过时开关
- 开关有默认值（安全优先）

### 风险 5：数据迁移过程中出现数据丢失
**概率**：低  
**影响**：高  
**缓解措施**：
- 迁移前进行完整备份
- 先在测试环境验证迁移脚本
- 分批次迁移，每次迁移后验证
- 迁移过程有详细日志
- 有回滚方案

## 🔍 重点实现细节

### 1. PlatformResourceRepository 实现细节
```java
public interface PlatformResourceRepository {
    // 基础查询
    List<PlatformResource> findAll();
    Optional<PlatformResource> findById(Long id);
    Optional<PlatformResource> findByCode(String code);
    List<PlatformResource> findByType(ResourceType type);
    
    // 一致性验证方法
    boolean validateConsistency();
    List<ConsistencyIssue> getConsistencyIssues();
    
    // 对比查询（同时查新旧表）
    ComparisonResult compareAll();
}
```

### 2. 数据一致性验证策略
- **实时验证**：每次查询时对比新旧结果（仅开发/测试环境）
- **定时验证**：每天凌晨 2 点全量验证
- **抽样验证**：生产环境对 1% 请求进行抽样验证
- **告警机制**：不一致率超过 0.1% 立即告警

### 3. MenuService 增强细节
```java
public interface MenuService {
    // 现有方法（继续使用）
    List<MenuItemVO> getTenantMenu(Long tenantId);
    
    // 新增：资源驱动的菜单（实验性质）
    List<MenuNode> getResourceDrivenMenu(Long tenantId);
    
    // 对比方法
    MenuComparisonResult compareMenus(Long tenantId);
}
```

### 4. 功能开关设计
- **配置文件开关**：`feature.unified-resources.read.enabled`
- **租户级开关**：数据库中 tenant_config 表配置
- **用户级开关**：特定用户可尝鲜（用于内部测试）
- **运行时切换**：支持不重启动态调整

### 5. 性能基准测试方法
- **测试工具**：JMH（Java Microbenchmark Harness）
- **测试场景**：
  - 查询所有资源
  - 根据类型查询资源
  - 获取租户菜单
- **对比指标**：吞吐量、延迟（P50/P95/P99）、内存使用
- **测试环境**：与生产配置尽可能一致

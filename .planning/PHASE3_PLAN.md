## 🛠️ 双写策略设计

### 1. 整体原则
- **旧系统优先**：旧表写入成功才认为操作成功
- **新表异步写入**：不影响主流程性能
- **失败告警与补偿**：新表写入失败时记录日志并告警，后续补偿
- **读操作优先新表**：新表读取失败时降级到旧表

### 2. TenantResourceConfigService 双写实现
```java
public interface TenantResourceConfigService {
    // 双写：同时写新表和旧表
    void updateResourceConfig(Long tenantId, String resourceCode, Object config);
    
    // 批量双写
    void batchUpdateResourceConfigs(Long tenantId, Map<String, Object> configs);
    
    // 读优先从新表，降级到旧表
    Map<String, Object> getTenantResources(Long tenantId);
    Optional<Object> getResourceConfig(Long tenantId, String resourceCode);
    
    // 一致性校验
    boolean checkConsistency(Long tenantId);
}
```

### 3. 双写流程
```
┌─────────────┐
│  业务请求   │
└──────┬──────┘
       │
       ▼
┌─────────────────┐
│  1. 写入旧表    │  ◀── 主流程，同步
└──────┬──────────┘
       │
       ▼
   旧表成功？───否───▶ 返回失败
       │
       是
       ▼
┌─────────────────┐
│ 2. 异步写入新表 │  ◀── 异步，不阻塞
└──────┬──────────┘
       │
       ▼
   新表成功？───否───▶ 记录日志、告警、加入补偿队列
       │
       是
       ▼
┌─────────────────┐
│   返回成功     │
└─────────────────┘
```

### 4. 补偿机制
- **定时任务**：每小时检查不一致数据并补偿
- **手动触发**：提供 API 手动触发补偿
- **补偿记录**：记录每次补偿操作，便于追踪

## 🔄 回滚机制设计

### 1. 回滚触发条件
- 新系统数据一致性低于 99.9%
- 性能下降超过 20%
- 业务功能异常
- 监控告警连续触发

### 2. 回滚步骤
1. **切换读写到旧系统**：通过 feature flag 关闭新系统
2. **验证旧系统正常**：检查关键业务功能
3. **数据对齐**：将新系统中的增量数据同步回旧系统（如需要）
4. **监控观察**：持续观察 24 小时确保稳定

### 3. 回滚保障
- **Feature flag 控制**：1 秒内切换
- **旧系统保持完整**：所有旧代码和表结构保留
- **数据备份**：回滚前自动备份关键数据
- **回滚演练**：灰度发布前进行回滚演练

## 📊 数据同步方案

### 1. 批量同步（初始数据迁移）
```sql
-- 从旧表批量导入到新表
-- 1. 导入平台资源
INSERT INTO platform_resources (...)
SELECT ... FROM products UNION SELECT ... FROM features;

-- 2. 导入套餐资源关联
INSERT INTO package_resources (...)
SELECT ... FROM package_products UNION SELECT ... FROM package_product_features;

-- 3. 导入租户资源配置
INSERT INTO tenant_resource_configs (...)
SELECT ... FROM tenant_products;
```

### 2. 增量同步
- **基于事件**：监听旧表变更事件，同步到新表
- **定时校验**：每 15 分钟校验一次数据一致性
- **差异修复**：发现差异自动修复或告警

### 3. 同步监控指标
- 同步延迟时间（目标：<5秒）
- 同步成功率（目标：>99.9%）
- 不一致记录数（目标：0）

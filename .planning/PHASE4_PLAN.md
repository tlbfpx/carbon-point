## 🚀 详细切换策略（金丝雀 → 50% → 100%）

### 1. 切换原则
- **渐进式切换**：从低比例开始，逐步扩大
- **可随时回滚**：每个阶段都保留完整回滚能力
- **监控驱动**：根据监控数据决定是否进入下一阶段
- **业务优先**：确保核心业务功能不受影响

### 2. Feature Toggle 配置
```yaml
# application.yml
feature:
  unified-resources:
    enabled: true           # 新架构总开关
    read-from-new: true     # 读路径：使用新系统
    write-to-new: true      # 写路径：使用新系统（新系统为主）
    legacy-system:
      enabled: true         # 保留旧系统（只读模式）
      read-only: true       # 旧系统只读，不接受写入
    canary:
      enabled: true         # 金丝雀发布开关
      percentage: 10        # 金丝雀流量比例
      tenants: []           # 指定租户列表（可选）
```

### 3. 切换阶段详解

#### 阶段 A：金丝雀发布（10%流量，第 1 周第 4 天）
- **目标**：验证新系统在生产环境的稳定性
- **策略**：
  - 按流量比例 10% 切换到新系统
  - 或者指定内部测试租户优先切换
  - 旧系统保持只读，作为 fallback
- **监控观察期**：24 小时
- **进入下一阶段条件**：
  - 错误率 < 0.1%
  - 性能满足或优于旧系统
  - 无核心业务功能异常
  - 数据一致性 100%

#### 阶段 B：50% 流量（第 2 周第 1 天）
- **目标**：验证新系统在中等负载下的表现
- **策略**：
  - 流量比例扩大到 50%
  - 继续监控关键指标
- **监控观察期**：24 小时
- **进入下一阶段条件**：
  - 错误率 < 0.1%
  - 性能稳定
  - 无新问题出现

#### 阶段 C：全量切换（100%流量，第 2 周第 2 天）
- **目标**：新系统完全接管
- **策略**：
  - 流量切换到 100%
  - 旧系统继续保留只读模式至少 1 周
- **监控观察期**：持续 72 小时

### 4. 切换执行步骤
1. **预切换检查**
   - 确认所有监控正常
   - 确认 Feature Flag 配置正确
   - 确认回滚方案准备就绪
   - 通知相关团队切换时间

2. **执行切换**
   - 通过配置中心动态调整 Feature Flag
   - 实时监控关键指标
   - 准备好随时回滚

3. **切换后验证**
   - 核心业务功能验证
   - 性能指标验证
   - 数据一致性验证

## 🔄 回滚演练程序

### 1. 回滚触发条件（满足任一即触发回滚）
- 核心业务功能异常，影响用户使用
- 错误率 > 1% 且持续 5 分钟以上
- 性能下降 > 30% 且持续 5 分钟以上
- 数据一致性校验失败
- 监控告警连续触发 P0 级别告警
- 安全漏洞被发现

### 2. 完整回滚演练场景

#### 演练场景：金丝雀发布阶段发现严重数据问题

**演练目标**：验证从 10% 流量回滚到旧系统的完整流程

**演练步骤**：

| 步骤 | 操作 | 负责人 | 预期结果 | 完成时间 |
|------|------|--------|----------|----------|
| 1 | 宣布回滚演练开始 | 吴敏 | 全体人员知晓 | T+0 |
| 2 | 触发回滚条件：模拟数据一致性告警 | 孙浩 | 监控系统发出 P0 告警 | T+1min |
| 3 | 收到告警，确认问题严重性 | 张伟、李娜 | 决策回滚 | T+5min |
| 4 | 执行回滚：修改 Feature Flag 配置 | 周杰 | 配置更新成功 | T+10min |
| 5 | 验证流量切回旧系统 | 王强 | 监控显示 100% 流量走旧系统 | T+15min |
| 6 | 验证旧系统功能正常 | 杨丽、孙浩 | 核心功能验证通过 | T+30min |
| 7 | 验证数据一致性 | 王强 | 数据校验通过 | T+45min |
| 8 | 确认回滚成功，宣布演练结束 | 张伟 | 系统恢复正常 | T+60min |
| 9 | 复盘总结 | 全体 | 形成改进报告 | T+2h |

### 3. 回滚详细步骤

#### 步骤 1：决策回滚
- 发现问题后，快速评估影响范围和严重程度
- 由架构师和项目经理共同决策是否回滚
- 通知所有相关人员

#### 步骤 2：执行回滚
```yaml
# 紧急回滚配置
feature:
  unified-resources:
    enabled: false          # 关闭新架构
    read-from-new: false    # 读路径切回旧系统
    write-to-new: false     # 写路径切回旧系统
    legacy-system:
      enabled: true
      read-only: false      # 旧系统恢复读写
    canary:
      enabled: false        # 关闭金丝雀
```
- 通过配置中心一键应用
- 预计时间：< 1 分钟

#### 步骤 3：回滚后验证
- 验证核心业务功能（登录、打卡、积分、兑换等）
- 验证数据一致性
- 验证性能指标
- 持续监控 24 小时

#### 步骤 4：数据对齐（如需要）
- 检查新系统中是否有旧系统未同步的数据
- 如有需要，将新系统的增量数据同步回旧系统
- 验证数据完整性

### 4. 回滚保障措施
- Feature Flag 配置热更新，无需重启服务
- 旧系统代码和表结构完整保留
- 回滚前自动备份关键数据
- 回滚流程文档化，团队成员熟悉
- 提前进行回滚演练

## 📊 性能压力测试计划

### 1. 压测目标
- 验证新系统性能不低于旧系统
- 确定系统瓶颈和容量极限
- 验证系统在高负载下的稳定性
- 性能改进目标：查询速度提升 ≥ 20%

### 2. 压测环境
- 环境：预发布环境（与生产配置一致）
- 数据量：模拟生产数据规模
- 工具：JMeter / Gatling

### 3. 压测场景

| 场景 | 并发用户 | 持续时间 | 描述 |
|------|----------|----------|------|
| 基准测试 | 50 | 10分钟 | 模拟正常负载 |
| 高峰负载 | 200 | 30分钟 | 模拟工作日高峰 |
| 极限测试 | 500 | 5分钟 | 测试系统极限 |
| 稳定性测试 | 100 | 4小时 | 长时间稳定性验证 |

### 4. 关键性能指标

| 指标 | 旧系统基准 | 新系统目标 |
|------|-----------|-----------|
| 菜单查询响应时间 | p95 < 200ms | p95 < 160ms (提升 20%) |
| 资源配置更新响应时间 | p95 < 100ms | p95 < 80ms |
| 首页加载响应时间 | p95 < 500ms | p95 < 400ms |
| 系统吞吐量 | 100 TPS | ≥ 120 TPS |
| 错误率 | < 0.1% | < 0.1% |

### 5. 压测执行计划
- **第 1 周第 2 天**：旧系统基准测试
- **第 1 周第 2-3 天**：新系统性能测试 & 优化
- **切换前**：最终性能验证

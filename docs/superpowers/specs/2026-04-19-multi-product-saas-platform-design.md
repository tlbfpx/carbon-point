# 多产品通用 SaaS 平台架构设计

> 日期: 2026-04-19
> 状态: Draft
> 范围: 概念设计 spec，不涉及代码改动

## 1. 背景与目标

### 1.1 现状

Carbon Point 目前是一个以"爬楼打卡送积分"为核心的多租户 SaaS 平台。系统已实现完整的积分规则引擎（时段规则、特殊日期翻倍、等级系数、连续打卡奖励）、权限套餐体系（free/pro/enterprise）、虚拟商品商城（券码/直充/权益）。

### 1.2 问题

平台绑定了"爬楼"这一单一产品形态，无法支持其他运动或行为激励场景（如走路、跑步、骑行）。每次新增场景都需要从零开发，缺乏产品抽象。

### 1.3 目标

将平台从单一产品泛化为支持多种产品的通用 SaaS 架构：

- 平台管理员可通过组合预置"积木"自助创建新产品
- 每个产品有独立的触发方式和业务规则，但共享积分账户、等级体系、兑换商城
- 套餐由产品集合组成，每个产品下有可选功能点，企业菜单权限由套餐决定
- 当现有积木不够用时，开发团队可扩展新的积木类型

### 1.4 Non-Goals

- 不涉及具体代码实现，本文档为概念设计
- 不改变多租户隔离方案（共享数据库 + tenant_id）
- 不改变等级体系定义和计算规则
- 不引入微服务拆分，保持 Spring Boot 单体应用

## 2. 整体架构

```
+----------------------------------------------------------+
|                     前端应用层                             |
|  platform-frontend  |  enterprise-frontend  |  h5        |
+----------------------------------------------------------+
|                     API 网关层                             |
|              统一入口 / JWT / 租户识别                      |
+----------------------------------------------------------+
|   产品层（可插拔模块）                                     |
|  +---------------+  +---------------+  +---------------+  |
|  | 爬楼积分       |  | 走路积分       |  | 未来产品...    |  |
|  | 触发器+规则    |  | 触发器+规则    |  | 触发器+规则    |  |
|  +-------+-------+  +-------+-------+  +-------+-------+  |
|          |                  |                  |           |
|  +------+------------------+------------------+------+    |
|  |           积木组件库（预置可复用）                    |    |
|  |  触发器: 打卡/传感器输入/...                          |    |
|  |  规则节点: 时段匹配/阈值/倍率/公式/上限/...           |    |
|  |  功能点: 连续奖励/趣味展示/特殊日期/...               |    |
|  +----------------------------------------------------+  |
+----------------------------------------------------------+
|   共享服务层（产品无关）                                   |
|  积分账户 | 等级系统 | 积分商城 | 数据报表 | 通知          |
+----------------------------------------------------------+
|   基础设施层                                              |
|  租户管理 | RBAC | 套餐-产品-功能点 | 文件存储 | 缓存      |
+----------------------------------------------------------+
```

### 分层原则

- **产品层**只知道自己的业务逻辑，不知道其他产品的存在
- **积木层**是预置的可复用组件，产品层按需组合
- **共享层**完全不感知产品差异，只接收"积分数"和"积分来源标记"
- **基础设施层**管理租户、权限、套餐等平台级能力

### 积分流转链路

```
[爬楼产品] --> +                  +--> 积分账户 (加/扣/冻结)
               |  "积分事件"      |--> 等级系统 (检查晋升/降级)
[走路产品] --> +-> PointsEventBus +--> 通知系统 (积分到账/等级变更)
               |                  |--> 积分流水记录
[未来产品] --> +
```

所有产品模块产出统一格式的积分事件，共享层不关心来源。

## 3. 套餐-产品-功能点数据模型

### 3.1 实体关系

```
平台级定义（无 tenant_id）                    企业级绑定
---------------------                    ---------------

permission_packages                      tenants
+-- id, code, name                       +-- id, name, ...
+-- ...                                  +-- package_id --> permission_packages.id
|
+-- package_products (套餐->产品, 多对多)
|   +-- package_id
|   +-- product_id --+
|   +-- sort_order   |
|                    |   product_configs (企业级产品配置, 有 tenant_id)
|   products         |   +-- id
|   +-- id  <--------+   +-- tenant_id
|   +-- code             +-- product_id
|   +-- name             +-- enabled (是否启用)
|   +-- trigger_type     +-- config_json (产品专属配置)
|   +-- icon             +-- ...
|   +-- description
|                        product_feature_configs (企业级功能点开关)
|   +-- product_features (产品->功能点)        +-- id
|   |   +-- product_id                        +-- tenant_id
|   |   +-- feature_id ----+                   +-- product_id
|   |   +-- is_required    |                   +-- feature_id
|   |       (必选/可选)     |                   +-- enabled
|   |                       |
|   product_features        |
|   +-- id  <---------------+
|   +-- code
|   +-- name
|   +-- feature_type (功能点类型标识)
|   +-- config_schema (功能点配置的 JSON Schema)
```

### 3.2 关系说明

| 关系 | 说明 |
|------|------|
| 套餐 -> 产品 | 多对多，一个套餐可包含多个产品 |
| 产品 -> 功能点 | 一对多，每个产品有自己的功能点列表 |
| 功能点.is_required | 必选功能点随产品自动开启，不可关闭 |
| 套餐 -> 功能点 | 间接关系：套餐包含产品，产品下的可选功能点由平台管理员在套餐中勾选 |
| 企业 -> 产品配置 | 企业绑定套餐后，可覆盖产品级配置（如步数系数） |
| 企业 -> 功能点开关 | 企业最终启用的功能点 = 套餐勾选的功能点 交集 企业级开关 |

### 3.3 示例

```
套餐 "标准版"
  +-- 产品: 爬楼积分
  |     +-- 时段规则配置 (必选)
  |     +-- 节假日翻倍   (可选, 套餐勾选 [x])
  |     +-- 周三送咖啡   (可选, 套餐勾选 [x])
  |     +-- 连续打卡奖励 (可选, 套餐勾选 [ ])
  +-- 产品: 走路积分
        +-- 步数积分换算 (必选)
        +-- 趣味等价物   (可选, 套餐勾选 [x])

企业A 购买"标准版" -> 菜单权限 = 所有已启用功能点对应的权限合集
```

## 4. 产品模块架构

每个产品模块由三部分组成：**触发器 + 规则链 + 功能点**。它们都从积木组件库中选取，产品模块负责组合。

### 4.1 积木组件接口

```
触发器 -- 决定用户如何触发积分
  getType(): String                    // 如 "check_in", "sensor_data"
  execute(ctx: TriggerContext): Result

规则节点 -- 链式处理积分计算
  getType(): String                    // 如 "time_slot", "threshold", "multiplier"
  process(ctx: RuleContext): Result    // 输入积分值，输出处理后的积分值

功能点 -- 独立于积分计算的业务功能
  getType(): String                    // 如 "consecutive_reward", "fun_display"
  getDefaultConfig(): FeatureConfig    // 功能点的默认配置结构
```

### 4.2 爬楼积分产品

```
产品: 爬楼积分 (code: stair_climbing)

触发器: CheckInTrigger
  "用户在时段内点击打卡按钮"

规则链: [时段匹配] -> [随机基数] -> [特殊日期倍率] -> [等级系数] -> [四舍五入] -> [每日上限]
  输入: 打卡时间
  输出: 最终积分值

功能点:
  +-- 时段规则配置 (必选, 规则链节点配置化)
  +-- 节假日翻倍   (可选, 对应规则链中的"倍率"节点)
  +-- 周三送咖啡   (可选, 特殊日期翻倍的一种预设)
  +-- 连续打卡奖励 (可选, 独立于规则链，在积分之后额外发放)
  +-- 积分兑换商品 (必选, 共享商城能力)
```

### 4.3 走路积分产品

```
产品: 走路积分 (code: walking)

触发器: SensorDataTrigger
  "用户每日领取一次，系统拉取当日步数"

规则链: [步数阈值过滤] -> [公式换算: floor(步数 x 系数)]
  输入: 当日步数
  输出: 最终积分值

功能点:
  +-- 步数积分换算   (必选, 规则链中的公式配置)
  +-- 趣味等价物展示 (可选, 独立展示逻辑)
  +-- 积分兑换商品   (必选, 共享商城能力)
```

步数数据通过手机健康 API（微信小程序 WeRun / APP HealthKit / Android Health Connect）持续采集，用户每天可领取一次积分。趣味等价物展示为企业可配置功能点，支持自定义等价物模板（如"相当于X根香蕉"、"相当于绕操场Y圈"）。

### 4.4 产品注册表

平台维护一个产品注册表，记录每个产品使用了哪些积木组件。Spring Boot 启动时，ProductRegistry 自动扫描所有 ProductModule Bean 并注册。

```
product_registry (代码级, 非 DB 表)
+-- stair_climbing
|     +-- trigger: CheckInTrigger
|     +-- rule_chain: [TimeSlotMatch -> RandomBase -> SpecialDateMultiplier -> LevelCoefficient -> Round -> DailyCap]
|     +-- features: [TimeSlotConfig, SpecialDate, WeeklyGift, ConsecutiveReward, PointsExchange]
|
+-- walking
      +-- trigger: SensorDataTrigger
      +-- rule_chain: [ThresholdFilter -> FormulaCalc]
      +-- features: [StepCalcConfig, FunEquivalence, PointsExchange]
```

设计要点：
- 规则链是**有序节点列表**，每个节点独立可测试
- 功能点和规则节点可以是一对一关系（如节假日翻倍 = 倍率节点的 UI 配置入口），也可以独立（如连续打卡奖励）
- "积分兑换商品"是跨产品的共享功能，在每个产品中标记为必选，实际由共享层统一处理
- 新产品 = 选择触发器 + 组装规则链 + 挑选功能点，在已有积木范围内不需要写代码
- 如果所需触发器或规则节点不存在（现有积木不够），提示"请联系开发团队添加新的组件类型"

## 5. 共享服务层

### 5.1 积分事件（标准消息）

所有产品模块产出统一格式的积分事件，共享层不关心来源：

```
PointsEvent {
    tenantId:    Long        // 租户ID
    userId:      Long        // 用户ID
    productCode: String      // "stair_climbing" / "walking"
    sourceType:  String      // "check_in" / "step_claim" / "consecutive_bonus" / "exchange"
    points:      Integer     // 正数=获得, 负数=消耗
    bizId:       String      // 业务单号（打卡记录ID / 兑换订单ID）
    remark:      String      // "早间打卡" / "步数领取 8232步" / "兑换咖啡券"
}
```

### 5.2 积分账户模型

沿用现有 users 表设计（total_points / available_points / frozen_points / level），积分流水表增加产品来源维度：

```
points_ledger（积分流水表，增加字段）
+-- id
+-- tenant_id
+-- user_id
+-- product_code      <- 新增：积分来源产品
+-- source_type       <- 新增：来源类型
+-- points            变动积分数
+-- balance_after     变动后余额
+-- biz_id            关联业务ID
+-- remark            描述
+-- created_at
```

### 5.3 等级系统

不变 -- 等级仍基于 total_points（所有产品积分汇总），等级晋升/降级规则和系数沿用现有设计。走路积分只是让用户多了一个攒积分的渠道。

### 5.4 积分商城

不变 -- 统一商城，用户不区分积分来自哪个产品。订单状态机沿用（pending -> fulfilled -> used/expired/cancelled）。

### 5.5 数据报表

报表需要按产品维度拆分：

```
企业数据报表
+-- 总览看板
|     +-- 总积分发放趋势（按天/周/月）
|     +-- 各产品积分占比（饼图：爬楼 60% / 走路 40%）
|     +-- 参与率（有领积分行为的人数 / 总用户数）
|
+-- 爬楼积分报表
|     +-- 打卡率趋势
|     +-- 时段分布
|     +-- 连续打卡天数分布
|
+-- 走路积分报表
|     +-- 日均步数趋势
|     +-- 步数区间分布（0-3k / 3k-6k / 6k-10k / 10k+）
|     +-- 领取率（达到阈值的用户中实际领取的比例）
|
+-- 兑换报表（不变）
      +-- 兑换量趋势
      +-- 热门商品排行
```

## 6. 平台管理后台

### 6.1 菜单结构

```
平台管理后台
+-- 企业管理
|     +-- 企业列表（名称、套餐、状态、到期时间、用户数）
|     +-- 企业详情（基本信息、绑定的套餐、各产品启用状态）
|     +-- 套餐变更（记录变更日志）
|     +-- 企业启用/停用
|
+-- 套餐管理
|     +-- 套餐列表
|     +-- 创建/编辑套餐
|     |     +-- 基本信息（名称、编码、描述）
|     |     +-- 选择产品（勾选本套餐包含哪些产品）
|     |     +-- 配置功能点（每个产品下勾选可选功能点）
|     +-- 套餐状态管理（启用/禁用）
|
+-- 产品管理
|     +-- 产品列表（所有已注册的产品模块）

|     |     +-- 基本信息（名称、编码、触发器类型、图标）
|     |     +-- 规则链预览（展示该产品使用的积木组件）

|     +-- 新建产品（从积木库中选取组合）
|     |     +-- 选择触发器
|     |     +-- 组装规则链（拖拽排序）
|     |     +-- 选择功能点

|
+-- 积木组件库
|     +-- 触发器管理（查看已有触发器类型）
|     +-- 规则节点管理（查看已有规则节点）

+-- 企业数据报表
|     +-- 平台总览（全平台企业数、用户数、积分发放量）

|
+-- 系统设置
      +-- 平台管理员账号管理
      +-- 通知模板管理
```

### 6.2 新建产品流程

```
Step 1: 基本信息
  +-- 输入产品名称、编码、描述
  +-- 选择触发器: [下拉选择已有触发器类型]

Step 2: 组装规则链
  +-- 左侧: 可用规则节点列表
  +-- 中间: 已选规则链（支持拖拽排序）
  +-- 右侧: 每个节点的参数配置面板

Step 3: 选择功能点
  +-- 可用功能点列表（勾选）
  +-- 标记每个功能点: 必选 / 可选

Step 4: 确认保存
  +-- 产品注册到平台，可被套餐引用
```

如果所需触发器或规则节点不存在（现有积木不够），提示"请联系开发团队添加新的组件类型"。

### 6.3 企业管理后台变化

企业管理后台的变化相对小 -- 菜单根据套餐动态生成。主要变化：

```
企业管理后台（根据套餐动态展示）
+-- 数据看板        (必有)
+-- 成员管理        (必有)
+-- 爬楼积分管理     <- 仅当套餐包含爬楼产品时显示
|     +-- 时段规则配置   <- 仅当功能点启用时显示
|     +-- 节假日翻倍配置
|     +-- 周三送咖啡配置
|     +-- 连续打卡奖励配置
+-- 走路积分管理     <- 仅当套餐包含走路产品时显示
|     +-- 步数换算配置   (系数配置)
|     +-- 趣味等价物配置
+-- 积分商城         <- 有任一产品即显示
+-- 积分管理         <- 有任一产品即显示
+-- 数据报表         <- 根据启用的产品展示对应报表页
```

### 7.2 模块依赖关系

```
carbon-app

  +-- carbon-report     -> carbon-points

```

### 7.3 产品模块启动自动注册

每个产品模块实现 ProductModule 接口并标注为 Spring Component。应用启动时 ProductRegistry 自动扫描所有 ProductModule Bean 并注册。平台管理后台可通过 ProductRegistry 查询所有已注册产品的元信息。

### 7.4 与现有代码的关系

| 现有模块 | 变化 |
|----------|------|
| carbon-checkin | 代码迁移到 carbon-stair，原模块可废弃 |
| carbon-points | 增加 PointsEventBus 和积分事件处理，流水表加 product_code 字段 |
| carbon-report | 增加 walk 相关报表和跨产品总览报表 |

## 9. 关键决策记录

### 决策 1：混合式产品架构

**选择**: 共享基础设施 + 产品独立逻辑模块 + 可复用积木组件库
**理由**: 匹配"平台管理员自助配置为主，开发扩展为辅"的需求。渐进式复杂度，初期只需实现爬楼和走路的积木，后续按需扩展。
**备选方案**:
- 纯元数据驱动（低代码引擎）: 过度抽象风险大，调试困难
- 纯代码插件: 每次需要写代码，无法自助创建

### 决策 2：统一积分池

**选择**: 所有产品积分汇入统一账户，统一商城兑换
**理由**: 用户体验简洁，不需要区分积分来源。积分流水保留 product_code 用于报表分析。
**备选方案**:
- 分账积分: 增加用户理解成本，兑换逻辑复杂

### 决策 3：产品注册表为代码级而非数据库表

**选择**: ProductRegistry 通过 Spring 组件扫描自动注册，不存储在数据库中
**理由**: 产品的触发器和规则链是代码逻辑，无法完全配置化。数据库只存储产品元信息和企业的启用配置。
**备选方案**:
- 全部数据库配置: 触发器和规则链的代码逻辑无法用配置表达

### 决策 4：走路积分每日领取制

**选择**: 步数持续采集，用户每日领取一次积分
**理由**: 与爬楼产品的"每日打卡"节奏一致，降低用户操作频率，简化防并发逻辑
**备选方案**:
- 实时多次领取: 增加并发复杂度，且步数数据本身是日汇总更准确
